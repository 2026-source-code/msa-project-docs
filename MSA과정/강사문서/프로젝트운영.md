## 1. PBL 프로젝트 계획서

### 1-1. 프로젝트 개요

| 항목 | 내용 |
|-----|------|
| 프로젝트명 | 1차 모놀로식 프로젝트 (자유주제) |
| 프로젝트 유형 | PBL(Project Based Learning) |
| 훈련시간 | 140시간 (약 4주) |

### 1-2. PBL 학습 목표

| 구분 | 학습 목표 |
|-----|----------|
| 기술 역량 | Spring Boot 기반 모놀로식 웹 애플리케이션 설계 및 구현 |
| 문제 해결 | 미학습 기술(Redis, SSE 등)을 자기주도적으로 학습하고 프로젝트에 적용 |

| 의사소통 | 하브루타 기반 토론, 멘토링, 발표를 통한 커뮤니케이션 능력 향상 |
| 실무 역량 | 요구사항 분석 → 설계 → 구현 → 테스트의 개발 사이클 경험 |

### 1-3. PBL 운영 원칙

> PBL(Problem/Project Based Learning)은 학생이 실제 문제를 중심으로 자기주도적으로 학습하는 방식입니다.
> 강사는 **퍼실리테이터(Facilitator)** 역할을 하며, 학생이 스스로 문제를 발견하고 해결하도록 안내합니다.

| 원칙 | 설명 |
|-----|------|
| 자기주도 학습 | 배우지 않은 기술도 직접 조사·학습하여 프로젝트에 적용 (예: Redis 세션, SSE, API 설계 등) |
| 문제 중심 | 실제 서비스를 만든다는 가정 하에 요구사항을 도출하고 문제를 해결 |
| 협력 학습 | 팀원 간 역할 분담, 코드리뷰, 하브루타 토론을 통한 상호 학습 |
| 반복적 피드백 | 주간 스프린트 단위로 멘토링 → 피드백 → 개선 사이클 반복 |
| 결과물 발표 | 중간/최종 발표를 통해 학습 내용을 정리하고 공유 |

### 1-4. PBL 단계별 운영 계획

| 단계 | 기간 | 활동 내용 | 강사 역할 | 산출물 |
|-----|------|----------|----------|--------|
| **1단계: 문제 인식** | 1~2일 (8h) | - 프로젝트 OT 및 PBL 방법론 안내<br>- 팀 구성 및 아이스브레이킹<br>- 자유주제 브레인스토밍<br>- 주제 선정 및 타당성 검토 | - PBL 방법론 설명<br>- 주제 방향성 가이드<br>- 팀 구성 지원 | - 팀 구성표<br>- 주제 선정서 |
| **2단계: 학습 계획 수립** | 3~5일 (16h) | - 요구사항 분석 및 정의<br>- 학습 필요 기술 목록 도출<br>- 개인별 학습 계획 수립<br>- 아키텍처 설계<br>- DB 설계 (ERD)<br>- 화면 설계 (와이어프레임) | - 요구사항 정의 방법 안내<br>- 아키텍처 설계 가이드<br>- 학습 자료 추천 | - 요구사항 정의서<br>- 학습 계획서<br>- 아키텍처 설계서<br>- ERD<br>- 화면 설계서 |
| **3단계: 자기주도 학습 및 구현 (Sprint 1)** | 6~10일 (40h) | - 미학습 기술 자기주도 학습<br>- 프로젝트 환경 설정<br>- 핵심 기능 구현 착수<br>- 일일 스탠드업 미팅<br>- 하브루타 노트 작성 | - 기술 질의응답<br>- 학습 방향 조언<br>- 관찰일지 기록 | - 프로젝트 기본 구조<br>- 핵심 기능 초안<br>- 하브루타노트 |
| **4단계: 중간 점검 및 피드백** | 11일 (8h) | - 중간 발표 (팀별 10~15분)<br>- 멘토 피드백<br>- 동료 평가<br>- 개선 계획 수립 | - 중간 평가<br>- 멘토링<br>- 관찰일지 기록 | - 중간 발표 자료<br>- 멘토링 피드백 기록<br>- 개선 계획서 |
| **5단계: 구현 고도화 (Sprint 2)** | 12~17일 (48h) | - 피드백 기반 개선<br>- 추가 기능 구현<br>- Redis 세션 관리 구현<br>- 코드리뷰 진행<br>- JPA 쿼리 테스트 | - 코드리뷰 참여<br>- 기술 이슈 해결 지원<br>- 관찰일지 기록 | - 구현 완료 코드<br>- 테스트 결과<br>- 코드리뷰 기록 |
| **6단계: 최종 발표 및 평가** | 18~20일 (20h) | - 최종 결과물 정리<br>- 발표 자료 준비<br>- 최종 발표 (팀별 15~20분)<br>- 프로젝트 회고 (KPT)<br>- 기술 블로그 정리 | - 최종 평가<br>- 종합 피드백<br>- 총평 작성 | - 최종 발표 자료<br>- 프로젝트 결과물<br>- 팀 기술 블로그<br>- 하브루타노트 최종본 |

### 1-5. PBL에서 학생이 자기주도 학습해야 하는 영역

> PBL 방식이므로 아래 기술은 수업에서 직접 가르치지 않으며, 학생이 프로젝트 진행 중 필요에 의해 스스로 학습합니다.

| 영역 | 자기주도 학습 항목 | 학습 참고 자료 |
|-----|-----------------|--------------|
| 세션 관리 | Redis를 활용한 세션 클러스터링 | Spring Session Redis 공식 문서 |
| 템플릿 엔진 | Mustache 문법 및 레이아웃 구성 | Mustache 공식 GitHub, Baeldung |
| 실시간 통신 | SSE(Server-Sent Events)를 활용한 알림 서비스 | Spring SseEmitter 공식 문서 |
| API 설계 | RESTful API 설계 원칙 및 적용 | REST API Tutorial, 기술 블로그 |
| 테스트 | JUnit5를 활용한 JPA Repository 쿼리 테스트 | JUnit5, @DataJpaTest 공식 문서 |
| 버전 관리 | Git Flow 전략, PR 기반 코드리뷰 | Atlassian Git Flow 가이드 |
| 보안 (선택) | Spring Security 기초, CSRF, XSS 방지 | Spring Security 공식 문서 |

### 1-6. 팀 구성 기준

| 항목 | 기준 |
|-----|------|
| 팀 규모 | 4~6명 |
| 구성 원칙 | 초급/중급/고급 수준의 훈련생 고르게 배치 |
| 역할 정의 | 팀장, 백엔드 리드, 프론트엔드 리드, DB 담당, QA/테스트 담당 |
| 주제 선정 | 팀별 브레인스토밍 후 자유주제 선정 (강사 승인 필요) |

### 2-4. 요구사항 우선순위 분류

| 우선순위 | 설명 | 예시 |
|---------|------|------|
| **필수 (Must)** | 프로젝트 핵심 기능, 없으면 서비스 불가 | 회원관리, 핵심 CRUD, Redis 세션, MVP 범위 기능 |
| **중요 (Should)** | 서비스 품질에 큰 영향, 가능한 포함 | 페이징, 예외처리, 유효성 검증 |
| **선택 (Could)** | 있으면 좋지만 없어도 기본 동작 가능 | 파일 업로드, SSE 알림, Spring Security, 검색 |
| **추후 (Won't)** | 이번 프로젝트에서는 제외, 2차 MSA에서 고려 | MSA 전환, 컨테이너화, CI/CD |
