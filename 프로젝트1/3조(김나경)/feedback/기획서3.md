# 캠핑장 예약 & 운영 관리 시스템 — 프로젝트 기획서 v3

> **프로젝트명:** CampEase (캠프이즈)
> **서비스 성격:** 단일 캠핑장의 예약 사이트 + 운영자 관리 시스템
> **팀 구성:** 4명 · **기간:** 4주 (Sprint 4회)
> **개발 방식:** Agile Scrum + AI 도구 활용 고속 개발
> **작성일:** 2026-02-26

---

## 1. 프로젝트 비전

**"전화 한 통 없이, 검색부터 퇴실까지 한 번에."**

하나의 캠핑장을 위한 전용 웹사이트.
고객은 사이트 정보부터 예약·문의·주문까지 **전화 없이 모두 해결**하고,
사장님은 **관리자 페이지 하나로** 예약·매출·고객·재고를 한눈에 관리한다.

---

## 2. 난이도 기준 (L1 ~ L5)

| 레벨 | 설명 | 예시 |
|---|---|---|
| **L1** | 정적 페이지, 단순 CRUD | 공지사항 게시판, 회원가입 폼 |
| **L2** | 기본 CRUD + 간단한 비즈니스 로직 | 리뷰 작성(별점+텍스트), FAQ 관리 |
| **L3** | 복합 비즈니스 로직, 상태 관리, 연관 테이블 다수 | 예약 신청+승인 흐름, 요금 자동 계산 |
| **L4** | 외부 API 연동, 실시간 처리, 복잡한 상태 전이 | 결제 연동, 날씨 API, 알림 시스템 |
| **L5** | 고급 알고리즘, AI 연동, 대시보드 시각화 | 매출 차트, 예약률 통계, 추천 알고리즘 |

## 3. 기술 스택

| 영역 | 기술 | 비고 |
|---|---|---|
| Backend | **Spring Boot 3.x (Java 17+)** | SSR 방식 — Controller에서 View 직접 반환 |
| Template Engine | **Thymeleaf** 또는 **JSP** | form 태그 기반 GET/POST 요청 처리 |
| 인증/인가 | **HttpSession + Interceptor** | Security 미사용, 세션 기반 로그인 유지 + 인터셉터로 권한 체크 |
| HTTP 메서드 | **GET / POST만 사용** | `<form>` 태그 `application/x-www-form-urlencoded` 기본 전송 |
| 예외 통신 | **Ajax (fetch/jQuery) + JSON** | form으로 불가능한 경우에 한해 최소한으로 사용 |
| DB (운영) | **MySQL 8.x** | 최종 테스트 및 시연 시 사용 |
| DB (개발) | **H2 Database** (인메모리) | 개발 중 빠른 테스트, application-dev.yml로 프로필 분리 |
| ORM | **Spring Data JPA + Hibernate** | 엔티티 기반 DB 매핑 |
| CSS | **Bootstrap 5** 또는 **Tailwind CSS** | 반응형 레이아웃 |
| 빌드 | **Gradle** | 의존성 관리 |
| AI 활용 | Cursor / Claude Code / Copilot | 코드 생성 가속 |
| 협업 | **Git + GitHub** | 브랜치 전략으로 협업, PR 기반 코드 리뷰 |
| 배포 | **없음 (로컬 실행)** | localhost에서 시연 |

### 프로젝트 구조 (SSR 방식)

```
src/main/java/com/campease/
├── controller/          ← @Controller (GET/POST 매핑, Model에 데이터 담아 View 반환)
├── service/             ← 비즈니스 로직
├── repository/          ← Spring Data JPA Repository
├── domain/              ← Entity 클래스
├── dto/                 ← 폼 데이터 바인딩용 DTO
├── interceptor/         ← 로그인 체크, 관리자 권한 체크 인터셉터
└── config/              ← WebMvcConfigurer (인터셉터 등록 등)

src/main/resources/
├── templates/           ← Thymeleaf 템플릿 (.html)
├── static/              ← CSS, JS, 이미지
├── application.yml      ← 공통 설정
├── application-dev.yml  ← H2 설정 (개발용)
└── application-prod.yml ← MySQL 설정 (운영용)
```

### 요청 처리 원칙

**원칙: form 전송 (x-www-form-urlencoded) 우선, Ajax는 최소한으로**

```
[기본] form 태그 전송 (95% 이상)
──────────────────────────────────────────────────────
<form method="POST" action="/reservation">    ← application/x-www-form-urlencoded
  → @Controller + @PostMapping
  → 처리 후 redirect:/결과페이지
  → 페이지 전체 새로고침 (PRG 패턴)

[예외] Ajax + JSON 통신 (최소한으로)
──────────────────────────────────────────────────────
fetch('/api/check-date', { method: 'POST', body: JSON.stringify(data) })
  → @RestController 또는 @ResponseBody
  → JSON 응답
  → 페이지 이동 없이 부분 갱신
```

### Ajax 사용 기준 (반드시 최소화)

> **화면 기획 시 Ajax가 최소한으로 들어갈 수 있도록 설계해야 한다.**
> form 전송으로 처리 가능하면 무조건 form 전송을 사용한다.

| 구분 | 설명 | 예시 |
|---|---|---|
| **form 전송 (기본)** | 페이지 이동이 있는 모든 요청 | 회원가입, 로그인, 예약 신청, 리뷰 작성, Q&A 작성, 관리자 승인 |
| **Ajax 허용 (예외)** | 페이지 이동 없이 즉시 결과가 필요한 경우만 | 아래 표 참고 |

**Ajax가 불가피한 경우 (예상)**

| 기능 | Ajax 필요 이유 | 통신 방식 |
|---|---|---|
| 예약 캘린더 날짜 선택 | 날짜 클릭 시 해당 날짜의 예약 가능 여부 즉시 확인 | `POST /api/check-availability` → JSON |
| 아이디 중복 확인 | 회원가입 중 페이지 이동 없이 즉시 확인 | `GET /api/check-username?id=xxx` → JSON |
| 부대용품 수량 변경 시 합계 | 수량 변경할 때마다 총 금액 실시간 계산 | JS 로컬 계산 (서버 통신 불필요) |
| 예약 요금 실시간 표시 | 인원·날짜 변경 시 예상 요금 즉시 표시 | `POST /api/calc-price` → JSON |

> **그 외 모든 기능은 `<form>` + `x-www-form-urlencoded` + redirect로 처리**

### 요청 흐름 (SSR)

```
[기본 흐름 — form 전송]
브라우저 ──form POST (x-www-form-urlencoded)──→ @Controller
  → Service → Repository → DB
  → redirect:/결과페이지 (PRG 패턴)
  → 브라우저가 GET으로 결과 페이지 요청
  → Model에 데이터 담기 → Thymeleaf → HTML 응답

[예외 흐름 — Ajax]
브라우저 ──fetch POST (application/json)──→ @RestController
  → Service → Repository → DB
  → JSON 응답
  → JavaScript로 화면 부분 갱신 (페이지 이동 없음)
```

### Git 협업 전략

```
main ─────────────────────────────────────── 배포 가능 상태 유지
  │
  ├── develop ────────────────────────────── 개발 통합 브랜치
  │     ├── feature/회원가입 ──→ PR ──→ merge
  │     ├── feature/예약신청 ──→ PR ──→ merge
  │     ├── feature/관리자-승인 ──→ PR ──→ merge
  │     └── feature/리뷰 ──→ PR ──→ merge
  │
  └── hotfix/긴급수정 ──→ main + develop에 merge
```

- **브랜치 네이밍:** `feature/기능명`, `fix/버그명`, `hotfix/긴급`
- **PR 필수:** develop 브랜치에 직접 push 금지, PR + 최소 1명 리뷰 후 merge
- **커밋 메시지:** `feat: 회원가입 기능 추가`, `fix: 예약 날짜 검증 오류 수정`

## 6. "전화 문의 제로" — 기능 상세 & 필요 데이터

기존 캠핑장에서 **전화로만 해결 가능했던 문제**를 온라인으로 전환하는 것이 핵심 차별점.
각 기능마다 **사장님(관리자)이 어떤 데이터를 입력해야 하는지** 함께 정리.

---

### 6-0. AI 챗봇 — LLM + MCP 기반 전화 문의 자동 대체 (#35)

#### 왜 AI 챗봇이 필요한가?

태그, FAQ, 사이트 스펙 등을 아무리 잘 정리해도 **고객은 자기 상황에 맞는 질문**을 한다:

```
"저희 4인 가족이고 대형견 1마리 있는데, 이번 주말에 갈 수 있는 사이트 있나요?"
"캠핑카(5.5m)로 가려는데 A존 들어가나요? 전기도 써야 하는데요"
"장작이랑 숯 미리 주문하면 사이트까지 갖다주시나요?"
```

이런 **복합 질문**은 FAQ로 커버 불가 → AI 챗봇이 DB 데이터를 조합해서 **즉시 답변**

#### LLM vs RAG vs MCP — 뭘 써야 하나?

| 방식 | 설명 | 우리 프로젝트에 적합한가? |
|---|---|---|
| **LLM 단독** | ChatGPT/Claude에게 캠핑장 정보를 프롬프트로 넣고 질문 | **부적합** — 정보가 바뀔 때마다 프롬프트 수정 필요, DB와 동기화 안 됨 |
| **RAG** | 캠핑장 정보를 벡터DB에 임베딩 → 질문 시 유사 문서 검색 → LLM에 전달 | **과하다** — 단일 캠핑장의 정형 데이터(사이트 스펙, 요금, 규정)는 벡터 검색보다 DB 직접 조회가 정확 |
| **MCP (Model Context Protocol)** | LLM이 우리 DB를 **직접 도구(Tool)로 조회** → 실시간 데이터 기반 응답 | **최적** — DB에 이미 모든 데이터가 있고, MCP로 LLM이 필요한 테이블을 직접 조회하면 됨 |

#### 결론: MCP 방식 채택

```
RAG가 필요한 경우:  비정형 데이터가 대량 (수천 개 문서, PDF, 블로그 등)
MCP가 적합한 경우:  정형 데이터가 DB에 이미 있음 ← 우리 프로젝트!
```

우리 캠핑장의 모든 데이터는 **이미 MySQL/H2 DB에 정형화**되어 있다:
- 사이트 스펙 → `site` 테이블
- 예약 현황 → `reservation` 테이블
- 요금 정보 → `season_price` 테이블
- 시설 태그 → `facility` 테이블
- 반려동물 규정 → `pet_policy` 테이블
- 부대용품 → `product` 테이블

**별도 벡터DB(RAG) 없이, MCP로 LLM이 DB를 직접 조회하면 끝.**

#### MCP 기반 챗봇 구조

```
고객 질문: "이번 주말에 대형견 데리고 갈 수 있는 사이트 있나요?"
     │
     ▼
[LLM (Claude API)]
     │
     ├── MCP Tool 호출: check_availability(날짜: 이번 주말)
     │     └── → reservation 테이블 조회 → 빈 사이트 목록 반환
     │
     ├── MCP Tool 호출: check_pet_policy(크기: 대형)
     │     └── → pet_policy 테이블 조회 → 대형견 가능 사이트 반환
     │
     └── 두 결과 교차 → "A-3, B-1 사이트가 가능합니다. 대형견 추가 요금 15,000원입니다."
```

#### MCP Tool 목록 (서버에서 제공)

| Tool 이름 | 기능 | 조회 테이블 |
|---|---|---|
| `get_site_info` | 사이트 상세 스펙 조회 | site, site_spec |
| `check_availability` | 날짜별 예약 가능 사이트 조회 | reservation, site |
| `get_price` | 사이트 요금 조회 (성수기/비수기 반영) | site, season_price |
| `get_pet_policy` | 반려동물 규정 조회 | pet_policy |
| `get_facilities` | 시설·서비스 목록 조회 | facility |
| `get_products` | 부대용품 목록·재고 조회 | product |
| `get_rules` | 이용 규칙 조회 | rule |
| `get_nearby` | 주변 편의시설 조회 | nearby_facility |
| `get_faq` | FAQ 조회 | faq |
| `get_weather` | 예약일 날씨 조회 | 외부 API 연동 |

#### 구현 방식 (Spring Boot + MCP)

```
[고객 브라우저]
     │
     │  Ajax (JSON) ← 챗봇은 Ajax 예외 허용
     ▼
[@RestController - /api/chat]
     │
     ▼
[MCP Server (Spring Boot 내장)]
     │
     ├── @Tool 어노테이션으로 각 Tool 등록
     ├── 각 Tool은 기존 Service/Repository 재사용
     └── LLM API (Claude) 호출 시 Tool 목록 전달
           │
           ▼
     [Claude API] ── Tool 호출 요청 ──→ [MCP Server] ──→ [DB 조회]
           │                                                │
           └──────── Tool 결과 수신 ◄──────────────────────┘
           │
           ▼
     자연어 답변 생성 → JSON 응답 → 브라우저 채팅 UI 표시
```

> **참고:** 챗봇 기능은 실시간 대화 특성상 **Ajax (JSON) 통신 예외 허용**

#### 챗봇 대화 예시

```
고객: 이번 주 토요일에 4명이서 가려는데, 바베큐 가능한 사이트 추천해주세요

챗봇: 이번 주 토요일(3/7) 예약 가능한 사이트 중 바베큐 가능한 곳을 찾았습니다!

      ✅ A-3 사이트 (데크) — 80,000원/박
         크기: 8m×10m | 주차: 2대 | 개별 화로대 제공
      ✅ B-1 사이트 (잔디) — 60,000원/박
         크기: 10m×10m | 주차: 1대 | 개별 화로대 제공

      참숯(8,000원), 장작(10,000원)도 예약 시 함께 주문 가능합니다.
      예약하시겠어요? 👉 [A-3 예약하기] [B-1 예약하기]
```

```
고객: 대형견(골든리트리버) 데려가도 되나요?

챗봇: 반려동물 동반 규정을 확인했습니다.

      🐕 대형견(25kg 이상): B존 사이트만 가능
         - B-1, B-2, B-3 사이트
         - 추가 요금: 15,000원/박
         - 규정: 목줄 필수, 배변봉투 지참, 공용시설 출입 제한

      방금 추천드린 B-1 사이트는 대형견 가능합니다! 😊
```

#### 난이도 & 공수

| 항목 | 난이도 | 비고 |
|---|---|---|
| MCP Tool 구현 (10개) | **L2** | 기존 Service/Repository 재사용, @Tool 어노테이션만 추가 |
| Claude API 연동 | **L3** | API 키 관리, 요청/응답 처리 |
| 채팅 UI (프론트) | **L3** | Ajax 기반 채팅 인터페이스, 메시지 표시 |
| **종합** | **L4** | MCP Tool 자체는 쉽지만 LLM 연동 + UI 통합이 중간 난이도 |

#### RAG가 필요 없는 이유 정리

| 비교 항목 | RAG | MCP (우리 선택) |
|---|---|---|
| 추가 인프라 | 벡터DB 필요 (Pinecone, ChromaDB 등) | **불필요** — 기존 MySQL/H2 그대로 사용 |
| 데이터 동기화 | 데이터 변경 시 재임베딩 필요 | **자동** — DB 직접 조회이므로 항상 최신 |
| 정확도 | 유사도 기반 → 부정확할 수 있음 | **정확** — SQL 조회 결과 그대로 |
| 적합한 데이터 | 비정형 (문서, PDF, 블로그) | **정형** (사이트 스펙, 요금, 예약 현황) ← 우리 데이터 |
| 개발 복잡도 | 임베딩 파이프라인 + 벡터DB 구축 | **낮음** — 기존 Service에 @Tool 추가 |
| 비용 | 임베딩 API 비용 추가 | LLM API 비용만 |

### 6-5. 얼리체크인 / 레이트체크아웃

## 14. 기존 캠핑장 사이트 이용 시 불편 사항 총정리

> 20년간 수많은 서비스를 기획하며 실제 캠핑장 이용자·운영자 양쪽에서 수집한 불편 사항들.
> 이 목록이 CampEase의 기능 설계 근거이며, 하나하나가 "왜 이 기능이 필요한가"에 대한 답변.

### A. 예약 과정의 불편

| # | 불편 사항 | 상세 | CampEase 해결 |
|---|---|---|---|
